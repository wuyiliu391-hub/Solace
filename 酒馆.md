# 酒馆功能计划

## 0. 当前结论

酒馆不是单一玩法，而是一个 **多角色场景系统**。

由于用户需求分化明显，第一版不把酒馆固定成某一种模式，而是提供多种模式，让用户可以在创建酒馆时选择，也可以在酒馆设置中手动切换。

核心目标：

- 让喜欢“多角色群聊”的用户能直接用。
- 让喜欢“剧情房间 / 角色扮演”的用户能沉浸使用。
- 让喜欢“AI 自己聊天、用户旁观”的用户也有玩法。
- 保持第一版稳定，不一次性接入过多复杂记忆、关系、自动剧情系统。

---

## 1. 功能定位

### 1.1 总定位

酒馆 = 多 AI 角色进入同一个场景后的互动空间。

它可以是：

1. 多角色群聊
2. 剧情场景房间
3. AI 自主聊天空间
4. 用户可随时介入的角色扮演现场

### 1.2 第一版设计原则

第一版优先解决：

- 入口清晰
- 创建流程清晰
- UI 稳定
- 交互可控
- 不误触高消耗自动模式
- 不让复杂 AI 逻辑影响主聊天稳定性

第一版暂时不追求：

- 完整长期剧情系统
- 复杂角色关系演化
- 深度多人记忆同步
- 无限自动聊天
- 高度复杂的 SillyTavern 级群聊管理

## 2. 酒馆模式设计

### 2.1 模式一：群聊模式

定位：最直观、最有“被一群人包围”归属感的模式。

交互：

- 用户发送一句话。
- 所有角色都应有机会回应。
- 默认按预设顺序轮流发言，而不是由系统替用户决定谁该闭嘴。
- 回复展示为普通群聊气泡。

适合用户：

- 想看多个 AI 一起回应自己。
- 想要热闹、陪伴、被围绕的感觉。
- 想把酒馆当多人聊天室。

优点：

- 群聊感最强。
- 用户能明确感受到“所有角色都在场”。
- 最符合“进入酒馆”的期待。

风险：

- 角色多时可能显得吵。
- token 消耗更高。

第一版建议：

- 群聊模式默认按顺序轮流发言。
- 不默认智能裁剪发言者。
- 如果用户怕吵，通过“沉浸度 / 安静模式”控制回复数量，而不是系统私自让角色沉默。

---

### 2.2 模式二：剧情模式

定位：沉浸式场景 / 角色扮演。

交互：

- 酒馆有一个明确场景，例如：雨夜咖啡馆、校园天台、深夜便利店、异世界旅店。
- 旁白优先，角色跟随。
- 旁白负责推进故事线，角色根据剧情需要出场。
- 这里不是谁想说话就说话，而是“剧本”说了算。
- 角色发言更像剧情文本。
- 用户可以像参与 RP 一样输入动作或对白。

示例：

```text
【雨夜咖啡馆】
小雪轻轻搅动杯中的热可可，看向窗外。

小雪：“雨好像一时半会停不了。”
林川：“那就多坐一会儿吧。”
你：“我刚好也不想走。”
```

适合用户：

- 喜欢剧情。
- 喜欢角色扮演。
- 喜欢场景氛围。

优点：

- 沉浸感强。
- 和普通单聊差异明显。

风险：

- 更依赖提示词质量。
- 容易跑偏。
- 输出可能变长，token 消耗更高。

第一版建议：

- 剧情模式先只作为 UI/提示词模式。
- 不做复杂剧情状态机。
- 提供几个固定场景模板 + 自定义场景。
- 每轮先生成旁白/场景推进，再根据需要安排 1~3 位角色出场。

---

### 2.3 模式三：旁观模式

定位：AI 角色自己聊天，用户可以旁观，也可以随时插话。

交互：

- 用户可以点击“让他们继续聊”，角色们会围绕上一个话题自己聊起来。
- 用户默认是看客，但可以随时发送消息插入对话。
- 目标是模拟真人社群的感觉：角色之间有来有回，而不是都只盯着用户。
- 不默认无限自动跑，避免失控和消耗过高。

适合用户：

- 想看角色之间互动。
- 想让 AI 自己产粮。
- 想偶尔插话而不是一直主导。

优点：

- 有生命感。
- 适合粉丝向玩法。

风险：

- token 消耗高。
- 容易变成无限生成。
- 角色之间可能抢话或重复。

第一版建议：

- 不做真正后台自动运行。
- 只做手动按钮：`让他们继续聊`。
- 每次生成有限条消息，具体数量受“沉浸度”控制。
- 生成内容以角色互相交流为主，而不是轮流向用户汇报。

---

### 2.4 模式四：点名模式

定位：高可控群聊。

交互：

- 用户可以 `@角色名`。
- 被点名角色优先回复。
- 其他角色最多补 1 条。
- 这条补充消息必须是和被点名角色互动，而不是抢着回应用户。
- 这样既不会冷场，也不会抢走被点名角色的风头。

适合用户：

- 想控制谁回答。
- 想避免多人乱回。
- 想做较精确的角色互动。

优点：

- 可控。
- 成本低。
- 不容易乱。

风险：

- 需要做好 @ 面板和角色选择交互。

第一版建议：

- 输入框支持 `@` 唤起角色列表。
- 或者先做简单按钮：点击角色头像 → “让 TA 回应”。

## 3. 发言规则与沉浸度

### 3.1 AI 互相接话

结论：必须允许，但必须可控。

酒馆区别于普通聊天软件的灵魂，不只是“用户和多个 AI 聊天”，而是“AI 之间也会互动”。用户来这里，是想看到一个真实的、有自己社交关系的小世界。

设计原则：

- 不禁止 AI 互相接话。
- 不让 AI 无节制互相刷屏。
- 提供“角色互动频率”控制。
- 不同模式使用不同默认频率。

建议设置项：

```text
角色互动频率
- 低：角色主要回应用户，偶尔互相接一句
- 中：角色会自然接话，有群聊感
- 高：角色之间互动频繁，像真实社群
```

### 3.2 沉浸度设定

Token 控制不要直接叫“省 Token 模式”，太技术、太冷。应翻译成用户能理解的“沉浸度”。

建议选项：

```text
轻聊模式：最多 1 人回复，适合只想和特定角色私聊
标准模式：1~2 人回复，适合日常陪伴
热闹模式：2~3 人回复，适合群聊氛围
狂欢模式：所有角色可回复，适合派对氛围
```

说明：

- 群聊模式下，狂欢模式允许所有角色按顺序轮流回复。
- 剧情模式下，沉浸度影响旁白长度和出场角色数量。
- 旁观模式下，沉浸度影响每次“让他们继续聊”生成的消息条数。
- 点名模式下，沉浸度不抢被点名者风头，最多允许其他角色补充互动。

### 3.3 各模式默认行为

```text
群聊模式：用户发一句，角色按预设顺序轮流回应；沉浸度决定本轮最多几人出声。
剧情模式：旁白优先推进剧情，角色根据剧情需要出场。
旁观模式：用户点击“让他们继续聊”，角色围绕上一个话题自主互动。
点名模式：@谁，谁优先回复；其他角色最多补 1 条，且必须和被点名角色互动。
```

## 4. 模式切换

### 3.1 创建时选择模式

创建酒馆时提供“酒馆玩法”选择：

1. 群聊模式
2. 剧情模式
3. 旁观模式
4. 点名模式 / 可控模式

但为了降低复杂度，第一版可以这样设计：

- 主模式三选一：群聊 / 剧情 / 旁观
- 点名能力作为所有模式都支持的辅助功能

### 3.2 酒馆内随时切换

酒馆详情页右上角设置中提供：

```text
酒馆模式
- 群聊
- 剧情
- 旁观
```

切换后影响：

- 输入提示文案
- AI 回复提示词
- 是否显示“让他们继续聊”按钮
- 消息展示风格

### 3.3 模式切换的安全限制

第一版不建议切换模式后改写历史消息格式。

也就是说：

- 历史消息保持原样。
- 新消息按新模式生成和展示。

## 5. UI 初步方向

### 4.1 消息页入口

当前入口：

```text
消息页 → 右上角 + → 创建酒馆
```

保留。

后续可考虑：

- 消息列表里普通聊天和酒馆混排。
- 酒馆卡片使用不同图标 / 多头像堆叠，避免看起来像普通单聊。

### 4.2 创建酒馆页

第一版创建页应包含：

1. 酒馆名称
2. 酒馆模式
3. 场景模板
4. 自定义场景
5. 选择角色，至少 2 位
6. 发言活跃度
7. 创建按钮

建议暂不放太多高级项，避免用户一进来就懵。

高级设置可以后置到“酒馆设置”。

### 4.3 酒馆详情页

建议使用混合样式：

- 顶部：酒馆名称 + 当前模式 + 成员头像
- 中间：聊天区
- 底部：输入框 + @按钮 + 继续按钮

不同模式下 UI 轻微变化：

#### 群聊模式

- 普通群聊气泡。
- 用户发言后 AI 自动回复。

#### 剧情模式

- 顶部显示场景卡片。
- 消息可偏剧本样式。
- 输入框提示：`输入对白或动作...`

#### 旁观模式

- 底部多一个按钮：`让他们继续聊`
- 用户不输入也能推进一小段对话。

## 6. 第一版暂定范围

### 5.1 必做

- 恢复创建酒馆入口。
- 创建页加入“模式选择”。
- 酒馆详情页显示当前模式。
- 消息列表中酒馆和普通聊天有明显区别。
- 基础群聊发送流程稳定。
- 支持至少 2 个角色。
- 创建失败 / 角色不足时有明确提示。

### 5.2 可做

- 场景模板优化。
- 角色头像堆叠。
- `@角色` 选择。
- `让他们继续聊` 按钮。
- 酒馆设置页调整模式。

### 5.3 暂不做

- 后台自动无限聊天。
- 深度关系演化。
- 酒馆记忆写入单人记忆。
- 多角色之间复杂长期关系。
- 高级 prompt 编辑器。

## 7. 仍需讨论的问题

### 7.1 发言细节

已确定大方向：

- 群聊模式：所有角色都有机会回应，按预设顺序轮流。
- 剧情模式：旁白优先，角色跟随剧情。
- 旁观模式：角色自主互动，用户是看客。
- 点名模式：被 @ 角色优先，其他角色最多补 1 条互动。

仍需细化：

- 预设顺序如何设置？按加入顺序、用户拖拽排序，还是随机轮换？
- 狂欢模式下角色很多时，是否设置单轮上限？
- 点名模式下“补 1 条”的触发条件是什么？
- 角色互动频率滑块具体分几档？

### 7.2 沉浸度与消耗控制

已确定：

- 不叫“省 Token 模式”。
- 改叫“沉浸度”。
- 档位为：轻聊、标准、热闹、狂欢。

仍需细化：

- 是否显示预计消耗？
- 是否给旁观模式加确认？
- 每个沉浸度对应的具体回复上限是多少？

### 7.3 记忆与关系

待定：

- 酒馆消息是否进入角色个人记忆？
- 酒馆是否有独立记忆？
- 酒馆里的角色关系是否会变化？
- 用户是否能手动保存某段酒馆事件？

### 7.4 UI 风格

待定：

- 普通群聊气泡为主？
- 剧本式文本为主？
- 是否允许用户切换展示样式？

## 8. 推荐开发顺序

### 阶段一：UI/交互骨架

- 创建页增加模式选择。
- 消息页酒馆卡片优化。
- 详情页顶部显示模式和场景。
- 加入“让他们继续聊”按钮，但先只做 UI 或简单事件。

### 阶段二：稳定群聊模式

- 用户发言后按预设顺序安排角色轮流回复。
- 沉浸度决定本轮最多几人出声。
- 狂欢模式允许所有角色可回复。
- 修复历史 bug。

### 阶段三：剧情模式

- 场景模板参与 prompt。
- 剧情模式消息展示优化。
- 自定义场景稳定写入。

### 阶段四：旁观模式

- `让他们继续聊` 按钮。
- 每次生成有限条数。
- 增加消耗提示。

### 阶段五：记忆/关系扩展

- 酒馆独立记忆。
- 可选写入角色记忆。
- 角色关系轻量变化。

---

## 9. 当前已确认

- 酒馆支持多种模式。
- 用户可以手动切换模式。
- 群聊模式默认让所有角色都有机会按顺序回应。
- 剧情模式以旁白推进为主，角色跟随剧本需要出场。
- 旁观模式以角色自主互动为核心，用户可以旁观或插话。
- 点名模式中，被 @ 角色优先，其他角色最多补 1 条且必须与被点名角色互动。
- AI 互相接话必须允许，但要通过“角色互动频率”控制。
- Token 控制包装为“沉浸度”：轻聊、标准、热闹、狂欢。
- 第一版先重 UI 和交互稳定性。
- 不做失控的后台自动聊天。
- 不急着接入复杂记忆和关系系统。
